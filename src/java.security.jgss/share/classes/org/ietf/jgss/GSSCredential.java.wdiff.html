  28   30  /**
  29   31   * This interface encapsulates the GSS-API credentials for an entity.  A
  30   32   * credential contains all the necessary cryptographic information to
  31   33   * enable the creation of a context on behalf of the entity that it
  32   34   * represents.  It may contain multiple, distinct, mechanism specific
  33   35   * credential elements, each containing information for a specific
  34   36   * security mechanism, but all referring to the same entity. A credential
  35   37   * may be used to perform context initiation, acceptance, or both.<p>
  36   38   *
  37   39   * Credentials are instantiated using one of the

  38   40   * {@code createCredential} methods in the {@link GSSManager
  39   41   * GSSManager} class. GSS-API credential creation is not
  40   42   * intended to provide a "login to the network" function, as such a
  41   43   * function would involve the creation of new credentials rather than
  42   44   * merely acquiring a handle to existing credentials. The
  43   45   * <a href=package-summary.html#useSubjectCredsOnly>section on credential
  44   46   * acquisition</a> in the package level description describes
  45   47   * how existing credentials are acquired in the Java platform. GSS-API
  46   48   * implementations must impose a local access-control policy on callers to
  47   49   * prevent unauthorized callers from acquiring credentials to which they
  48   50   * are not entitled. <p>
  49   51   *
  50   52   * Applications will create a credential object passing the desired
  51   53   * parameters.  The application can then use the query methods to obtain
  52   54   * specific information about the instantiated credential object.
  53   55   * When the credential is no longer needed, the application should call
  54   56   * the {@link #dispose() dispose} method to release any resources held by
  55   57   * the credential object and to destroy any cryptographically sensitive
  56   58   * information.<p>
  57   59   *
  58   60   * This example code demonstrates the creation of a GSSCredential
  59   61   * implementation for a specific entity, querying of its fields, and its
  60   62   * release when it is no longer needed:
  61   63   * <pre>
  62   64   *    GSSManager manager = GSSManager.getInstance();
  63   65   *
  64   66   *    // start by creating a name object for the entity
  65   67   *    GSSName name = manager.createName("myusername", GSSName.NT_USER_NAME);
  66   68   *
  67   69   *    // now acquire credentials for the entity
  68   70   *    GSSCredential cred = manager.createCredential(name,
  69   71   *                    GSSCredential.ACCEPT_ONLY);
  70   72   *
  71   73   *    // display credential information - name, remaining lifetime,
  72   74   *    // and the mechanisms it has been acquired over
  73   75   *    System.out.println(cred.getName().toString());
  74   76   *    System.out.println(cred.getRemainingLifetime());
  75   77   *
  76   78   *    Oid [] mechs = cred.getMechs();
  77   79   *    if (mechs != null) {
  78   80   *            for (int i = 0; i{@literal <} mechs.length; i++)
  79   81   *                    System.out.println(mechs[i].toString());
  80   82   *    }
  81   83   *
  82   84   *    // release system resources held by the credential
  83   85   *    cred.dispose();
  84   86   * </pre>
  85   87   *
  86   88   * @see GSSManager#createCredential(int)
  87   89   * @see GSSManager#createCredential(GSSName, int, Oid, int)
  88   90   * @see GSSManager#createCredential(GSSName, int, Oid[], int)
  89   91   * @see #dispose()
  90   92   *
  91   93   * @author Mayank Upadhyay
  92   94   * @since 1.4
  93   95   */
